# AI Workflow (Marketplace Backend)

Кратко: какими AI-инструментами пользовались, как строили итерации, какие проверки выполняли, ключевые инженерные решения и их причины (§8.1).

---

## 1. Какими AI-инструментами пользовались

- **Cursor** — генерация кода, рефакторинг, тесты, миграции, конфиги.
- **Роль инженера:** ставит задачу и контракты, проверяет результат (тесты, ручные сценарии), вносит точечные правки и доводит до production-ориентированного уровня. Код не принимается «как есть» без проверки.

---

## 2. Как строили итерации: план → генерация → проверка → исправление

| Этап | План | Генерация (AI) | Проверка | Исправление |
|------|------|----------------|----------|-------------|
| **Архитектура** | Стек: FastAPI, SQLAlchemy 2.x async, PostgreSQL, MinIO, Alembic, JWT. | Структура папок, модули, конфиг (pydantic-settings). | Ревью структуры, соответствие ТЗ. | Уточнение границ модулей. |
| **Модели** | Продукты, атрибуты, продавцы, офферы; UUID, связи. | Модели SQLAlchemy 2.x, миграция Alembic (async). | `alembic upgrade head`, сверка схемы. | Правка типов/связей по необходимости. |
| **Публичный API** | Список товаров (пагинация), детали (атрибуты, офферы с seller name/rating, сортировка, default price). | Роутеры, join с Seller, Query default. | Ручные запросы, smoke-тесты. | Добавление полей, фикс сортировки. |
| **Admin API** | Auth (JWT), CRUD продуктов/офферов, загрузка изображения. | Роутеры auth, products, offers, media. | Логин, CRUD, upload вручную. | Исправление контрактов и зависимостей. |
| **Инфраструктура** | MinIO, seed, Dockerfile, .env. | Клиент storage/, seed (Faker), .env.example. | Запуск через compose, проверка endpoint’ов. | Правка переменных и путей. |
| **Тесты** | Smoke: list, detail 404, login 401. | test_smoke.py, conftest.py. | `pytest tests/ -v`. | Фикс тестов при изменении API. |

Каждая итерация — отдельный запрос к AI с чёткой задачей; результат проверяется перед переходом к следующему шагу.

---

## 3. Какие проверки выполняли

- **Тесты:** smoke-тесты (`pytest tests/ -v`): GET /v1/public/products (200, items), GET product по несуществующему UUID (404), POST /v1/admin/auth/login с неверными данными (401). Запуск при поднятой БД (локально или в Docker).
- **Линтеры / типы:** проверка импортов и типов в коде; соответствие слоям (модели, схемы, роутеры).
- **Ручные сценарии:** запуск приложения (`uvicorn` или через docker compose), миграции, seed; проверка /docs, запросы к публичному каталогу и admin (логин, создание продукта/оффера, загрузка изображения).

Исправления задаются точечными промптами (например: «вернуть seller_name и seller_rating в офферах», «сделать сортировку по умолчанию по цене»).

---

## 4. Ключевые инженерные решения и их причины

| Решение | Причина |
|---------|---------|
| **Async везде (asyncpg, AsyncSession, async-роуты)** | Масштабируемость и соответствие рекомендациям FastAPI/SQLAlchemy 2.x. |
| **Разделение: модели / schemas / роутеры; MinIO в `storage/`** | Понятная структура, изоляция хранилища, проще тестировать и менять реализацию. |
| **Конфиг через pydantic-settings, префикс `APP_`, .env.example без секретов** | Безопасность (секреты не в репо), единая точка конфигурации, удобство деплоя (в т.ч. через Docker). |
| **Placeholder-изображение (1×1 GIF) при отсутствии картинки** | Всегда валидный ответ по URL изображения; реальные картинки — через Admin upload. |
| **Smoke-тесты в одном файле, без моков БД** | Быстрая проверка «поднимается ли API и контракт» при наличии тестовой БД. |

Материалы позволяют убедиться, что код создавался и эволюционировал управляемо, с контролем качества (тесты, линтеры, ручные сценарии) — критерий приёмки §8.2.
